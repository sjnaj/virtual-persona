🏗️ 系统架构总览

她是唯一的个体——一个大脑、一份情绪、一段人生。但她在不同聊天窗口面对不同的人时，会自然地切换语气和边界，就像真人在微信里和不同的人聊天一样。

💡 核心设计原则：紫色区域是她的"大脑"，全局唯一；粉色区域是她对每个人的"印象"，各自独立；蓝色区域是跨场景共享的智能，但带有隐私标记。渐变边框的 InnerStateManager 是 v2.1 新增的内心独白层，负责情绪跨日延续和跨对话跟进。

🧩 模块详解

每个模块都是可独立理解的组件。点击查看详情。

🎭

Orchestrator 中枢协调器

整个系统的指挥中心。接收所有外部消息，协调各模块协同工作，管理后台循环任务。它知道"她"在做什么、跟谁说话、该不该回。

orchestrator.py
🌙

LifeSimulator 生活模拟

模拟她的24小时日常：起床、上班、吃饭、下班、刷手机、睡觉。包括体力、饥饿度、位置信息。影响她的可用性和回复速度。

life_simulator.py
💛

EmotionEngine 情绪引擎

PAD情绪模型（愉悦度-激活度-支配度）+ 依恋系统。情绪会影响她说话的语气、回复速度、表情包选择，甚至她是否想主动找人聊天。v2.1 新增 daily_baseline 支持情绪衰减到非零基线。

emotion_engine.py
💭

InnerStateManager 内心状态

v2.1 新增。她的"潜意识层"——每隔2-3小时生成一段内心独白，维护"脑子里挂着的事"（PendingThought），并在起床时用昨天的状态计算今天的情绪基线。让她有跨对话连续性和不可预测的情绪底色。

inner_state.py v2.1 NEW
👥

RelationshipManager 关系系统

每个人在她心里都有一份独立的"印象档案"：亲密度、信任度、标签、常聊话题、雷区、inside jokes。关系会随互动自然演化。

relationship.py
🧠

MemoryHub 记忆中枢

三层记忆：情节记忆、语义记忆、情感印记。记忆带隐私标记（来自私聊还是群聊）。支持自然遗忘、模糊化和情感加权。v2.1 起在 consolidate 完成后 emit memory.consolidated 事件。

memory_hub.py
💭

ChatContextManager 上下文管理

每个聊天窗口独立的对话流管理。维护最近消息历史、会话活跃状态、群成员列表。区分私聊和群聊的上下文。

chat_context.py
🎯

GroupBehaviorEngine 群聊行为

决定她在群里何时说话、怎么说。考虑发送者亲密度、话题匹配、情绪、性格外向性、连续沉默时长等因素。模拟真人的群聊行为模式。

group_behavior.py
✍️

ExpressionSynthesizer 表达合成

最终输出层。综合情绪、关系、记忆、上下文、生活状态，生成人格化的消息。v2.1 起 prompt 中新增"心里在转的"区块，融入内心独白。

expression.py
📱

BrowserAgent 浏览代理

模拟她刷手机：RSS、微博热搜、感兴趣的内容。浏览到的内容可能在聊天中自然提起，但不会强行安利。

browser_agent.py
😸

StickerEngine 表情包引擎

表情包的收集、分类和使用。根据心情和语境选择表情包。表情包是她个性的重要体现。

sticker_engine.py
💬

ProactiveEngine 主动引擎

触发主动发消息的时机判断：看到有趣内容想分享、想念某人、到了特定时间等。v2.1 起新增 follow_up 触发类型，由 PendingThought 驱动。

proactive_engine.py
🔌

EventBus 事件总线

模块间的发布/订阅通信机制。解耦各模块的依赖关系。v2.1 新增事件：life.woke_up, inner_state.updated, proactive.follow_up_fired, memory.consolidated。

event_bus.py

🔄 数据流

一条消息从收到到回复的完整旅程。

STEP 1 — 触发评估

ProactiveEngine.evaluate()

定期（每5分钟）检查是否有主动发消息的触发条件。来源包括：看到有趣内容想分享、想念某人（依恋度高 + 久未联系）、到了特定时间点（早安/晚安）、情绪需要倾诉、以及 v2.1 新增的 follow_up 触发器——由 PendingThought 驱动的"想跟进某件事"。

STEP 2 — 选择目标

选择发给谁

按亲密度排序，选择最适合的对象。对于 follow_up 类型，目标由 PendingThought.target_user_id 直接指定——她记得要找谁说这件事。其他类型按原逻辑选择。

STEP 3 — 生成消息

Expression.compose_proactive()

和普通回复的生成流程相同，但 prompt 中标明是"主动想说"而非"回复对方"。内容更随意：分享链接、吐槽、日常碎碎念、或者跟进之前的话题。

STEP 4 — 发送 & 清理

通过已注册的回调发送

orchestrator 为每个聊天窗口注册了 proactive_callback。触发后直接发送到对应的 Telegram 聊天。v2.1：若触发类型为 follow_up 且发送成功，emit proactive.follow_up_fired 事件，InnerStateManager 自动将对应的 PendingThought 从列表中移除。

## 6个后台循环任务，持续运行 Loop 1: _life_tick_loop # 每15分钟 → LifeSimulator.tick() # 更新作息/体力/位置 → EmotionEngine.passive_decay() # 情绪自然衰减（v2.1:衰减到daily_baseline而非0） Loop 2: _browse_loop # 每60分钟 → BrowserAgent.should_browse() # 判断是否要刷手机 → BrowserAgent.browse() # 抓取RSS/热搜 → EventBus → emotion_change # 浏览内容影响情绪 Loop 3: _proactive_loop # 每5分钟 → ProactiveEngine.evaluate() # 是否主动发消息（含follow_up触发器） → → 选择目标 → Expression → 发送 Loop 4: _memory_loop # 每6小时 → MemoryHub.consolidate() # 缓冲区→长期记忆 → MemoryHub.forget_and_distort() # 自然遗忘+扭曲 → emit memory.consolidated # ★ v2.1: 通知InnerState Loop 5: _conversation_end_check # 每2分钟 → 检测30分钟无互动的窗口 # 标记会话结束 → 触发该窗口的记忆整合 # 及时巩固 Loop 6: _inner_state_loop # ★ v2.1 NEW: 每2~3小时（随机间隔） → if is_sleeping: skip # 睡着了不生成独白 → InnerStateManager.generate() # 收集上下文 → LLM → 独白快照 → 更新 monologue + pending_thoughts # 清理过期thoughts，新增新的 → emit inner_state.updated # 通知proactive_engine等 → 持久化到 data/inner_state.json # 重启不丢失 ## 事件驱动（不是loop，由EventBus触发） Event: life.woke_up # ★ v2.1: 起床时触发 → InnerStateManager.on_wake_up() # 用昨天状态计算今日mood_baseline → EmotionEngine.set_daily_baseline()# 设置情绪衰减地板 Event: life.sleeping # ★ v2.1: 入睡时触发 → 快照 emotion.valence → yesterday_final_valence → 持久化

💭 内心状态系统

v2.1 新增。解决两个让人格显得"机器味"的核心缺陷：跨对话没有连续性，以及情绪过于稳定可预测。

✨ InnerStateManager 是她的"潜意识层"——她会在脑子里默默琢磨事情、情绪会从昨天延续到今天、她会记得要跟某个人聊某件事。这些让她不再像每次对话都是全新启动的机器。

解决的两个问题

❌ 问题 1：跨对话没有连续性

她不会主动跟进之前提过的事，每次对话像是全新开始。relationship.py 的 unresolved 字段存在但从未被填充或消费。

❌ 问题 2：情绪过于稳定可预测

emotion_engine 的衰减是纯数学的，缺少无由来的坏心情、隔天情绪延续、以及"今天就是心情不好"这类真实的情绪底色。

三大支柱

🫧

内心独白 Inner Monologue

每隔 2~3 小时（随机间隔），她会生成一段"内心独白"——对当前状态、最近发生的事、心里挂着的事的一小段自言自语。这段独白会被注入到 expression.py 的 prompt 中，自然地影响她说话的语气和内容。

"最近总有点心不在焉，也不知道为什么…上次小明说的那个项目后来怎样了，改天问问他吧。"

独白的 mood_tint 仅供 prompt 注入，不与 EmotionEngine 的 daily_baseline 互换。

📌

悬而未决 Pending Thoughts

她脑子里"挂着"的事——想问某人某件事、想跟进某个话题、想分享某个发现。每条 PendingThought 绑定目标用户，有紧急度和过期时间（默认48小时）。这是跨对话连续性的核心载体。

📌 "想问小明那个项目后来怎样了" → target: 小明 · urgency: 0.6
📌 "昨天看到一首歌想分享给你" → target: 你 · urgency: 0.4

过期自动遗忘，发送成功后自动移除，也可在独白生成时被 LLM 标记为 resolved。

🌅

情绪跨日延续 Mood Continuity

起床时，今天的情绪不再从零开始——它由昨天的最终情绪状态、随机扰动、星期权重和睡眠质量共同决定。这让她有"今天就是莫名心情不好"的真实感。

今日 baseline = 昨日 × 0.4 + 随机(±0.15) + 星期权重 + 睡眠影响

起床情绪公式

# InnerStateManager.on_wake_up() — 由 life.woke_up 事件触发 today_baseline = yesterday_final_valence × 0.4 # 延续感，但不完全继承 + random.uniform(-0.15, +0.15) # 无由来的情绪波动 + weekday_weight # 周一 -0.05，周五 +0.05，周末 +0.03 + (-0.1 if sleep_debt > 30 else 0) # 睡眠债影响 # 然后调用： emotion_engine.set_daily_baseline(today_baseline) # 此后 passive_decay() 中 valence 衰减到 daily_baseline 而非 0： # valence = baseline + (valence - baseline) × 0.98 # arousal 和 irritability 继续按原逻辑衰减到 0

边界条件

🆕 首次启动

yesterday_final_valence = 0，daily_baseline = 0，随机扰动正常工作

❌ LLM 调用失败

保留上一次 monologue，不更新 pending_thoughts

⏰ Thought 过期

每次生成独白时清理 expires_at < now 的条目（默认48h）

😴 睡眠中

跳过独白生成。入睡时快照 valence 到 yesterday_final_valence

⚠️ mood_tint ≠ daily_baseline：独白的 mood_tint 是主观色调，仅供 expression prompt 使用；daily_baseline 是 EmotionEngine 的衰减地板，仅在起床时设置一次。两者是不同概念，不可互换。

👥 群聊引擎详解

真人在群里不是每条消息都回的。她会潜水、偶尔冒泡、被@了才认真回、和熟人互动更多。

回复概率计算

# 基础概率从0开始，逐项叠加 base_prob = 0.0 # 1. 被@了 → 100%回复（硬规则） if mentioned_me: return 1.0 # 2. 发送者亲密度 base_prob += closeness × 0.25 # 好朋友(0.8)→+0.20, 陌生人(0.1)→+0.02 # 3. 话题匹配兴趣 base_prob += 0.15 # 每匹配一个兴趣关键词 # 4. 性格外向性 base_prob += extraversion × 0.08 # 外向的人更爱在群里说话 # 5. 情绪影响 if mood_good: base_prob += 0.08 if mood_bad: base_prob -= 0.10 # 6. 连续沉默修正（不能一直潜水） if silence > 15: base_prob += 0.10 if silence > 30: base_prob += 0.20 # 7. 刚说过话（防刷屏） if 刚说完 < 2min: base_prob ×= 0.20 # 8. 有人接自己话 → +0.15 # 9. 消息含问号/邀请 → +0.08 # 最终 prob ∈ [0, 0.85]

三种回复模式

🎯 Direct

正经回复消息内容。多条消息、表情包都可能出现。适用于被@、话题高度相关、和亲近的人互动。

😂 Reaction

轻回应——"哈哈"、"+1"、一个表情包、一句简短感叹。适用于概率偏低但还是想参与的场景。

🔗 Piggyback

借话题展开自己想说的。看到别人聊某个话题，顺势分享自己的经历或看法。很自然的群聊行为。

🔐 隐私边界系统

她知道什么话是在哪里听到的，不会在群里说出别人私聊里的秘密。

🔓 Public 记忆

来自群聊或公开场合的信息。可以在任何场景中自然提起。

"A在群里说他最近在学画画" → source_privacy: "public" → 可以在任何场景提起

🔒 Private 记忆

来自私聊的信息。在群聊中检索到时，标记为 private_hints——"你知道但不该说"。

"A私聊跟我说他要辞职" → source_privacy: "private" → 群里有人问A近况时： prompt: "⚠️ 你知道但不该说" 她的回复: "我也不太清楚诶"
⚠️ 隐私系统的效果取决于LLM的遵从能力。在prompt中明确标注"不要在群里说出来"后，大模型通常能很好地遵守。但建议在高敏感场景中额外做关键词过滤检测。

启动后的初始化流程

STEP 1

启动 Bot，获取 bot_username

系统启动后自动获取 bot 的用户名，用于群聊中的 @检测。

STEP 2

加载已有的关系、记忆和内心状态

从 data/relationships.json、data/chroma/ 和 data/inner_state.json 加载持久化数据。第一次启动为空，daily_baseline 默认为 0。

STEP 3

启动后台循环

❓ 常见问题

Q: 她在群里会不会说太多/太少？

▼

GroupBehaviorEngine 控制回复概率。你可以通过调整 persona.personality.extraversion 来改变她在群里的活跃度。外向值0.8的她会比0.3的她在群里发言多很多。你也可以直接修改 should_respond() 中的概率公式。

Q: 两个AI聊天会不会无限循环？

▼

会的。建议在群聊场景中添加anti-loop机制：检测到对方是bot时降低回复概率（如×0.3）、限制每5分钟最多回复同一bot 2次、或者在回复之间加入更长的随机延迟。在 handle_message 中可以通过 Telegram 的 is_bot 字段判断。

Q: LLM 费用大概多少？

▼

每次回复调用1次 expression LLM（prompt约2000 token + 回复约200 token）。如果使用 GPT-4o-mini，大约 $0.001/次。关系深度反思（每20次对话1次）和记忆整合（每6小时1次）各额外调用1次 utility LLM。v2.1 的内心独白生成每2~3小时1次，使用 utility tier LLM，额外增加约 $0.5~1/月。日常聊天量约100条/天的话，月费用大约 $4-6（GPT-4o-mini）或 $16-28（GPT-4o）。

Q: 记忆数据库会不会越来越大？

▼

forget_and_distort() 会定期清理低权重记忆（accuracy < 0.15 的自动删除）。情感权重高的记忆保留更久（就像人对情感强烈的事印象更深）。实际使用中，大约每3个月会稳定在 2000-5000 条记忆左右。如果需要更激进的清理，可以调低遗忘系数。

Q: 可以不用 Telegram，换成其他平台吗？

▼

可以。Orchestrator 是平台无关的。bot.py 只是 Telegram 的适配层。你可以写一个 discord_bot.py、wechat_bot.py、或者 web_api.py，只要调用 orchestrator.handle_message() 并传入正确的参数即可。核心的关系/记忆/情绪/内心状态系统完全不依赖 Telegram。

Q: 她知道自己是AI吗？

▼

在prompt中明确指示"不要说作为AI"。但这不是万无一失的——如果用户直接问"你是AI吗"，LLM 可能会出戏。建议在 persona 设定中加入对这种问题的预设反应（比如"开玩笑的吧？""你说呢？"等回避策略）。

Q: 内心独白会暴露给用户吗？ ★ v2.1

▼

不会。独白内容仅在系统内部流转——注入到 expression.py 的 prompt 中影响语气和内容，但不会直接输出给用户，也不会出现在 /status 命令中。用户只会感受到"她今天说话的感觉不太一样"，而不会看到"内心独白：xxx"这样的文字。这是设计上的刻意选择。

Q: PendingThought 会不会太频繁地触发主动消息？ ★ v2.1

▼

不会。PendingThought 的 follow_up 触发器与现有的 missing / mood_share / life_event 触发器并存，共同竞争 ProactiveEngine 的发送机会。ProactiveEngine 本身有频率限制（每5分钟评估一次，但不是每次都会发）。而且如果 target_user_id 对应的用户没有注册 proactive_callback，该 thought 会被跳过。过期时间默认48小时，可以在 config.yaml 中调整。
