# Media Input Feature Design

**Date:** 2026-03-22
**Status:** Approved (v2)

## Overview

Add the ability for the bot to receive and visually understand user-sent photos and static stickers via base64-encoded image content passed to the vision-capable expression LLM. Animations, videos, and animated/video stickers are gracefully declined with a persona-appropriate reply generated directly in `bot.py` (no LLM call required).

## Supported Media Types

| Telegram Type | Handling | Notes |
|---|---|---|
| `photo` | Download largest size → base64 JPEG | Mime type hardcoded as `image/jpeg` (Telegram `PhotoSize` has no mime field) |
| `sticker` (static `.webp`) | Download → base64 WEBP | `sticker.is_animated == False and sticker.is_video == False` |
| `sticker` (animated `.tgs`) | Hardcoded polite decline in `bot.py` | TGS is Lottie JSON, no rasterizable frame accessible via Bot API |
| `sticker` (video `.webm`) | Hardcoded polite decline in `bot.py` | `sticker.is_video == True` |
| `animation` (GIF/MP4) | Hardcoded polite decline in `bot.py` | Telegram serves animations as MP4; first-frame extraction requires heavy dependency (opencv/ffmpeg); not worth it |
| `video` | Hardcoded polite decline in `bot.py` | LLM does not support raw video |
| `video_note` (circle video) | Hardcoded polite decline in `bot.py` | Treated same as video |
| `document` (image mime) | Out of scope | Not implemented |
| `voice` / `audio` | Out of scope | Not implemented |

**Media count cap:** A single message produces at most **1 image** sent to the LLM. Telegram media album (multiple photos in one message) is not handled in this feature — only the first photo in the update is used; if the update contains multiple images, take the largest `PhotoSize` of the first `photo` field only (standard Telegram single-photo behavior; albums arrive as separate updates).

## Polite Decline Reply

Generated statically in `bot.py` — **no orchestrator/LLM call needed**. Use a random pick from a small fixed list to avoid repetition:

```python
_DECLINE_MEDIA = [
    "这个我看不了哎～",
    "视频/动图我这边显示不出来哈哈",
    "嗯这个我打开不了，发图片给我吧",
]
```

This is intentionally not persona-generated by the LLM to save tokens and keep latency low.

## Data Structure

A `media` field is threaded through the call chain as `list[dict] | None`:

```python
{
    "mime_type": "image/jpeg",  # or image/webp
    "base64": "<base64_string>",
    "label": "photo"            # photo | sticker
}
```

- `mime_type` and `base64` are consumed by `llm.call_vision()`
- `label` is consumed by `orchestrator.py` (memory text annotation) and `expression.py` (prompt annotation); `llm.py` does not use it

## Architecture

### Supported Media Call Chain

```
Telegram photo/sticker update
    ↓
bot.py: _handle_media_message()
    - _is_allowed() check
    - is_bot guard
    - mentioned_me detection (caption + reply_to)
    - _download_and_encode(file_id, label) → {mime_type, base64, label}
    ↓
orchestrator.py: handle_message(text=caption, ..., media=[...])
    - appends "[图片]" / "[表情包]" to stored memory/context text
    ↓
expression.py: compose_reply(..., media=[...])
    - annotates "对方发来" section with image count/label
    - calls llm.call_vision(prompt, images=media, tier="expression",
                            system_prompt=system_prompt, temperature=0.85)
    ↓
llm.py: call_vision(prompt, images, tier, system_prompt, ...)
    - builds OpenAI multimodal content list
    ↓
Claude Sonnet 4.6 (vision)
```

### Declined Media Flow

```
Telegram animation/video/video_note/animated-sticker update
    ↓
bot.py: _handle_decline_media_message()
    - _is_allowed() check
    - is_bot guard
    - send hardcoded decline reply immediately
    ↑ no orchestrator/LLM involved
```

## File-by-File Changes

### `llm.py`

New method `call_vision`:

```python
async def call_vision(
    self,
    prompt: str,
    images: list,           # [{mime_type, base64, label}, ...]
    tier: str = "expression",
    system_prompt: str = "",
    temperature: float = 0.85,
    max_tokens: int = 2000,
) -> str:
```

Implementation:
- Build `content` as a list: text node first, then one `image_url` node per image:
  ```python
  {"type": "image_url", "image_url": {"url": f"data:{img['mime_type']};base64,{img['base64']}"}}
  ```
- If `images` is empty or `None`, fall back to `call(prompt, tier=tier, system_prompt=system_prompt, temperature=temperature, max_tokens=max_tokens)` — the caller's `tier` and `max_tokens` are preserved unchanged
- Retry logic: same 3-attempt pattern as `call()`
- The `label` field in each image dict is ignored by this method

### `bot.py`

New private helpers:

```python
async def _download_and_encode(self, bot, file_id: str, label: str) -> dict | None:
    """Download a Telegram file and return base64 media dict. Returns None on error."""

async def _handle_media_message(self, update: Update, context: ContextTypes.DEFAULT_TYPE):
    """Handle photo and static sticker messages."""
    # 1. Guard: _is_allowed() → silent return
    # 2. Guard: is_bot → silent return (no reply sent)
    # 3. Extract caption text (may be empty string "")
    # 4. Detect mentioned_me from caption text (@username) and reply_to_message
    # 5. Download and encode the file → media list (None on error → proceed with media=None)
    # 6. Call orchestrator.handle_message(text=caption or "", media=media_list, ...)
    # 7. Send replies

async def _handle_decline_media_message(self, update: Update, context: ContextTypes.DEFAULT_TYPE):
    """Handle animation, video, video_note, animated/video sticker — decline politely."""
    # 1. Guard: _is_allowed() → silent return
    # 2. Guard: is_bot → silent return (no reply sent)
    # 3. Group-chat suppression: detect mentioned_me (same caption/@username + reply_to logic);
    #    if chat_type in ("group", "supergroup") and not mentioned_me → silent return
    # 4. Otherwise: send one random entry from _DECLINE_MEDIA
```

**`mentioned_me` detection for media messages** (same logic as text handler):
- Check caption text for `@{self.bot_username}`
- Check `update.message.reply_to_message.from_user.id == self.bot_id`

**File selection:**
- `photo`: use `update.message.photo[-1]` (largest available size)
- `sticker` (static): use `update.message.sticker.file_id`

**Sticker routing:**
```python
if update.message.sticker:
    if update.message.sticker.is_animated or update.message.sticker.is_video:
        # → decline
    else:
        # → media handler (static webp)
```

Handlers registered in `run()`:
```python
# Static stickers and photos → vision handler
app.add_handler(MessageHandler(
    (filters.PHOTO | filters.Sticker.STATIC) & ~filters.COMMAND,
    self._handle_media_message,
))
# Everything else that needs decline
app.add_handler(MessageHandler(
    (filters.VIDEO | filters.ANIMATION | filters.VIDEO_NOTE
     | filters.Sticker.ANIMATED | filters.Sticker.VIDEO) & ~filters.COMMAND,
    self._handle_decline_media_message,
))
```

### `orchestrator.py`

`handle_message()` signature:

```python
async def handle_message(
    self,
    text: str,
    user_id: int,
    user_name: str,
    chat_id: int,
    chat_type: str = "private",
    group_title: str = "",
    mentioned_me: bool = False,
    media: list = None,          # NEW
) -> Optional[list]:
```

Memory/context text label (unified string used in both places):

| `label` value | Text appended to stored message |
|---|---|
| `"photo"` | `" [图片]"` |
| `"sticker"` | `" [表情包]"` |

Both `memory.add_message(...)` and `chat_ctx.add_message(...)` get this annotated text. The raw `media` list is passed to `expression.compose_reply()`.

### `expression.py`

`compose_reply()` and `_compose()` signature:

```python
async def compose_reply(
    self,
    user_message: str,
    user_id: int,
    chat_id: int,
    chat_type: str = "private",
    reply_mode: str = "direct",
    media: list = None,          # NEW
) -> List[dict]:
```

In `_compose()`, the "对方发来" section is extended:

```python
if media:
    labels = "、".join(m["label"] for m in media)
    media_note = f"（附带了 {labels}）"
else:
    media_note = ""

user_content_line = (
    f'"{user_message}"{media_note}' if user_message
    else f"（{media_note.strip('（）')}，无文字）"
)
```

LLM call (replacing the current `self.llm.call(...)`):

```python
raw = await self.llm.call_vision(
    prompt,
    images=media or [],
    tier="expression",
    system_prompt=system_prompt,   # ← explicitly forwarded
    temperature=0.85,
    max_tokens=2000,               # ← explicit, symmetric with fallback path
)
```

`call_vision` falls back to plain text when `images=[]`, so no branching needed here.

## Error Handling

| Scenario | Behavior |
|---|---|
| Download failure (network/timeout) | Log warning; call orchestrator with `media=None` (text-only reply) |
| File > 20 MB (Telegram bot API limit) | `get_file()` raises exception; caught, treated same as download failure |
| Unsupported mime type returned | Log debug; skip encoding; treat as download failure |
| `call_vision` LLM failure | Same 3-attempt retry inherited from base `call()`; returns `""` on final failure; `expression.py` fallback to `"嗯嗯"` |

## Memory & Relationship Impact

Binary media is never stored. Memory and chat-context layers receive the text label suffix (e.g., `"好漂亮的猫 [图片]"`). This keeps ChromaDB text-based and avoids corrupting semantic embeddings.
